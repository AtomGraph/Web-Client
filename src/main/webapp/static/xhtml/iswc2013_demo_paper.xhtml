Introduction

Database technologies have been evolving very rapidly in the last decade. Programming language design has also advanced, and there is growing focus on functional languages, which have better scalability and parallelism properties compared to traditional imperative languages. However, the architecture of Web applications was designed with relational databases and object-oriented paradigms in mind, the combination of which suffers from object-relational impedance mismatch. With Graphity, we propose a declarative approach to Web applications, which maps read-write Linked Data HTTP access to SPARQL operations. The W3C has done related work based on XML, but has not applied it to RDF. The Linked Data Platform has similar aims, but is more general and does not force the mapping to SPARQL.

We demonstrate that the declarative approach has multiple advantages:

  1. moving code logic to data makes applications homoiconic, i.e. data can be managed using standard RDF tools and embedded in ontologies
  2. the processor is generic, its code base is stateless and functional
  3. declarative applications can run on different processors and platforms, can be imported from remote locations, merged, forked, managed collaboratively, transformed, queried etc.

The Graphity processor is open-source and works with any SPARQL 1.1 triplestore, while the commercial platform layer provides multi-tenant functionality such as access control and faceted search. It has been successfully used to build rich Linked Data applications for product information management and library data, featuring interactive maps, faceted and text search, autocomplete and complex content creation and editing for authenticated users.

Approach

We propose a Linked Data application architecture that specifies how Linked Data HTTP access maps to SPARQL access on RDF stores. The configuration of both the Linked Data resources and the SPARQL queries is left to the application. In RESTful applications, resources that share the same URI structure usually share the same representation pattern. In our case, the representation is RDF, defined by SPARQL. This allows us to achieve a mapping between resource URIs and SPARQL queries.

Implementation

All URI requests are first matched with an associated SPARQL query before fetching the data. The results may also be matched with a representation template before sending out the response.

Applications are declaratively configured. There is a simple regex-like JAX-RS syntax for URI templates, and a SPIN RDF syntax for SPARQL queries, we can define the mapping declaratively in an RDF ontology or vocabulary. The application might also need to produce representation types other than RDF. For example, (X)HTML output can be simply achieved via RDF/XML and XSLT on server and client-side.

Writing RDF data can also be implemented in the same declarative fashion, using templates, OWL class restrictions, and SPIN data quality constraints. The last piece of the declarative read-write Linked Data loop is retrieving user input natively as RDF. Luckily, that is covered by RDF/POST which provides a way to encode RDF as standard HTML form inputs.

Using Graphity Java implementation, application instance embeds the processor, which executes the RDF description on request. No domain-specific object layer is present: triplestore acts as MVC model via HTTP, and components operate on RDF natively and directly.
